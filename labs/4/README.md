Sequential Scan (последовательное сканирование):
- Библиотекарь ищет по порядку, с 1-й книги до последней
- Проверяет каждую запись
- Время: O(n) — пропорционально количеству строк
- Для 1 млн строк = долго

Index Scan (поиск по индексу):
- Как оглавление в книге → сразу на нужную страницу
- Библиотекарь видит: "нужная цена в диапазоне 500-2000" → идёт только туда
- Время: O(log n) — логарифмически, намного быстрее
- Для 1 млн строк = мгновенно

Порядок важен! PostgreSQL использует индекс слева направо:
- Запрос WHERE price > 500 → индекс работает ✅ (первое поле)
- Запрос WHERE category = 1 → индекс работает ❌ (нужно price)
- Запрос WHERE price > 500 AND category = 1 → работает ✅


Пример: каталог в библиотеке
- Сначала сортировка по авторам (фамилия)
- Потом по году внутри каждого автора
Если ищешь "все книги Иванова за 2020" — работает
Если ищешь "все книги за 2020" (без автора) — не работает, потому что внутри автора сортировано по году
Поэтому порядок полей в индексе важен: левая часть должна использоваться в запросе.

Bitmap — это про производительность
Когда находишь 500,000 строк (половина таблицы):
- Index Scan сделал бы 500,000 отдельных обращений к диску → медленно
- Bitmap: собирает все номера строк в один список → идёт за ними за один проход → быстро


1.  Основной список (таблица equipment): Это все ваши товары, их названия, цены, количество и т.д. Они лежат как попало, не отсортированы. Чтобы найти нужный товар, нужно просмотреть этот список от начала до конца.
2.  Индекс (указатель): Это отдельный, маленький и ОЧЕНЬ хорошо отсортированный список, который для каждой цены (price_per_day) говорит: "Вот тут лежат товары с такой ценой".
---
Обычный "Index Scan" (Индексное сканирование)
Допустим, вы ищете товары с ценой 1500.
1.  Вы смотрите в индекс: "Где товары по 1500?"
2.  Индекс даёт вам список из, скажем, 500 ID товаров.
3.  Вы берёте первый ID, идёте в основной список, находите товар.
4.  Берёте второй ID, идёте в основной список, находите товар.
5.  ... и так 500 раз.
Это быстро, если нужно найти 1-10 товаров. Но если их 500, или 10 000 — каждое "сходить в основной список" занимает время. Представьте, что вы 10 000 раз бегаете туда-сюда.


"Bitmap Index Scan" (Битовое Индексное Сканирование)
А теперь вы ищете товары с ценой ОТ 500 ДО 2000. Таких товаров может быть 100 000!
Bitmap Index Scan работает в два шага:
Шаг 1: Создаём "черновик" (Bitmap)
1.  Вы смотрите в индекс: "Где все товары от 500 до 2000?"
2.  Индекс выдаёт вам огромный список ID товаров. Вместо того, чтобы сразу бежать за каждым товаром, PostgreSQL создаёт специальную "карту" в памяти.
3.  Эта карта — как гигантский лист бумаги с миллионом клеточек (по числу строк в таблице). Если строка с таким-то ID нужна, мы ставим в её клеточке "галочку" (это и есть "бит").
4.  Итак, индекс просто быстро проставляет галочки на этой карте для всех 100 000 нужных ID. Он не бегает к основному списку!
Шаг 2: Идём за "галочками" (Bitmap Heap Scan)
1.  Теперь у нас есть карта с 100 000 галочек.
2.  PostgreSQL смотрит на эту карту и понимает, какие строки нужны. Он сортирует эти ID по порядку их физического расположения на диске.
3.  Затем ОДИН раз он идёт в основной список (таблицу) и забирает все 100 000 товаров сразу, оптимальным способом, потому что знает, где они лежат. Ему не нужно бегать туда-сюда 100 000 раз, он делает один большой "рейс".
В чём прикол и почему это "магия":
*   Меньше беготни: Вместо того, чтобы каждый раз обращаться к медленному диску за каждой строкой, мы сначала собираем все "адреса" в памяти, а потом идём за ними одним большим блоком. Это как составить список покупок, а потом пойти в магазин один раз, а не бегать в него за каждым продуктом.
*   Удобно для нескольких условий: Если у вас два условия (price BETWEEN 500 AND 2000) AND (category_id IN (1,2,3)):
    1.  Индекс по цене сделает свою битовую карту.
    2.  Индекс по категории сделает свою битовую карту.
    3.  PostgreSQL просто "накладывает" эти две карты друг на друга, ищет, где галочки совпали (операция AND/OR на битах).
    4.  И только потом идёт за строками, которые соответствуют ОБЕИМ картам.
Это делает сложные запросы с несколькими условиями по разным полям гораздо быстрее.



Что такое "индекс" в базе данных?
В базе данных индекс — это отдельная, специально структурированная копия (или часть копии) данных из одной или нескольких колонок таблицы, которая хранится в отсортированном виде.
Ключевые моменты:
*   Отдельная структура: Индекс — это не сама таблица. Это дополнительный объект, который база данных создаёт и поддерживает.
*   Сортировка: Данные в индексе всегда отсортированы по значению индексируемой колонки (или колонок). Именно эта сортировка позволяет быстро искать.
*   Указатели на строки: Рядом с каждым отсортированным значением в индексе хранится "указатель" (pointer) на физическое расположение полной строки в основной таблице. Этот указатель похож на "номер страницы" в аналогии с книгой. Он говорит: "Вот это значение находится в такой-то строке таблицы".
*   Цель: Ускорить поиск и сортировку данных, не перечитывая всю таблицу целиком.

Зачем он нам нужен?
*   Быстрый поиск: Когда ты делаешь WHERE city = 'Москва', база данных может быстро найти "Москва" в отсортированном индексе по колонке city и сразу получить указатели на все строки, где city равно "Москва".
*   Быстрая сортировка: Когда ты делаешь ORDER BY price_per_day DESC, если есть индекс по price_per_day (особенно DESC), база данных может просто прочитать этот индекс, потому что он уже отсортирован.


Пример CREATE INDEX idx_equipment_price ON equipment (price_per_day);
Что происходит, когда ты создаешь такой индекс:
1.  База данных берёт все значения из колонки price_per_day таблицы equipment.
2.  Создаёт новый файл (или добавляет в существующий индексный файл) эти значения.
3.  Сортирует их: 100, 100, 101, 102, ..., 9999, 10000.
4.  Рядом с каждым значением price_per_day сохраняет "адрес" той полной строки в таблице equipment, к которой это значение относится.
5.  В итоге получается быстрая "справочная таблица" вида:
    *   price_per_day = 100 -> адрес строки 1, адрес строки 500, адрес строки 999...
    *   price_per_day = 101 -> адрес строки 2, адрес строки 501, адрес строки 1000...
    *   ...и так далее.
Теперь, когда ты ищешь WHERE price_per_day BETWEEN 500 AND 2000, база данных не читает 1 млн строк таблицы equipment. Она читает только небольшой, отсортированный индекс, находит в нём все цены в нужном диапазоне, собирает их "адреса", а потом по этим адресам быстро забирает полные строки из основной таблицы.
индекс — это не просто список ID, а упорядоченная структура данных (как алфавитный указатель), которая содержит значения из колонки и ссылки на полные строки таблицы

B-tree Index (B-дерево)
Как работает:
Представь себе большую, хорошо организованную библиотеку. B-tree — это как главный каталог библиотеки, отсортированный по алфавиту. Чтобы найти книгу, ты не перебираешь все полки. Ты открываешь каталог, он направляет тебя к нужной секции, потом к нужной полке, и ты быстро находишь книгу. B-tree постоянно делит данные на поддиапазоны, сужая область поиска.
Плюсы:
1.  Универсальность: Подходит для большинства типов запросов.
2.  Диапазонный поиск: Отлично работает с операторами сравнения (>, <, >=, <=, BETWEEN). Например, "найти все товары с ценой от 500 до 2000".
3.  Сортировка: Ускоряет запросы с ORDER BY, так как данные в индексе уже отсортированы.
4.  Поиск по префиксу: Эффективен для LIKE 'начало%' (поиск по началу строки).
5.  Композитные индексы: Можно создавать индексы по нескольким колонкам, и они будут работать, пока запрос использует поля слева направо.
Минусы:
1.  Больше операций при записи: Каждая операция INSERT, UPDATE, DELETE требует перебалансировки дерева, что занимает время. Чем больше индексов, тем медленнее запись.
2.  Неэффективен для неточного поиска: Для LIKE '%часть%' (поиск по середине или концу строки) B-tree не помогает, приводит к полному сканированию.
3.  Медленнее для точного равенства на низкой кардинальности: Если значений очень мало и они повторяются часто (например, колонка gender со значениями "муж" и "жен"), B-tree может быть не самым быстрым для WHERE gender = 'муж', так как всё равно придётся пройти по веткам дерева.
---
Hash Index (Хеш-индекс)
Как работает:
Представь, что у тебя есть куча почтовых ящиков, и у каждого ящика свой уникальный номер. Когда приходит письмо, ты вычисляешь "хеш" (уникальный код) для адреса получателя, и этот хеш говорит тебе, в какой почтовый ящик положить письмо. Чтобы найти письмо, ты снова вычисляешь хеш адреса и идёшь сразу к нужному ящику. Хеш-индекс берёт значение колонки, пропускает через хеш-функцию и получает уникальный "ключ", который указывает на место хранения данных.
Плюсы:
1.  Очень быстрый поиск по точному равенству: Идеален для запросов типа WHERE status = 'Завершен' или WHERE user_id = 123. Скорость практически не зависит от размера таблицы.
2.  Меньше места на диске: Обычно занимает меньше места, чем B-tree для тех же данных.
Минусы:
1.  Только точное равенство: Главный минус — хеш-индексы работают ТОЛЬКО с оператором =. Они абсолютно бесполезны для >, <, BETWEEN, LIKE, ORDER BY.
2.  Нет сортировки: Данные не хранятся в отсортированном виде, поэтому не ускоряют ORDER BY.
3.  Не поддерживают композитные индексы: Нельзя создать хеш-индекс по нескольким колонкам, чтобы он работал как :составной.
4.  Коллизии: Хотя и редко, но разные значения могут дать одинаковый хеш (коллизия), что может потребовать дополнительных проверок и немного замедлить поиск.
---
Когда что использовать?
*   B-tree:
    *   Твой выбор по умолчанию для большинства случаев.
    *   Когда тебе нужен поиск по диапазону (BETWEEN, >, <).
    *   Когда нужна сортировка (ORDER BY).
    *   Когда ты ищешь по префиксу (LIKE 'часть%').
    *   Когда тебе нужны композитные индексы.
*   Hash:
    *   Очень специфический выбор для колонок, по которым ты ищешь только по точному равенству, и эти колонки имеют низкую кардинальность (мало уникальных значений, много повторений, как status или gender), или когда тебе нужна максимальная скорость для =.
    *   Используется значительно реже B-tre







- про pg_trgm
pg_trgm — это расширение (extension) для PostgreSQL, которое предоставляет функции и операторы для работы с триграммами.
Что такое триграмма?
Триграмма — это последовательность из трёх соседних символов в строке.
Пример:
Слово "лыжи" может быть разбито на триграммы следующим образом:
*   _лы (символ _ часто используется как начало/конец слова)
*   лыж
*   ыжи
*   жи_
---
Зачем нужны триграммы и pg_trgm?
Основное назначение pg_trgm — ускорение и улучшение полнотекстового и подстрочного поиска, особенно когда стандартные B-tree индексы неэффективны.
Например, для запросов типа:
*   WHERE name LIKE '%лыжи%' (поиск по середине/концу слова)
*   WHERE description ILIKE '%быстрый%' (регистронезависимый поиск)
*   Поиск похожих слов (fuzzy matching)
Почему B-tree плох для таких запросов?
Как мы уже обсуждали, B-tree индекс хранит слова отсортированными от начала. Если ты ищешь %лыжи%, базе данных нужно просмотреть всю таблицу (Sequential Scan), потому что она не знает, с какой "середины" начинается твой шаблон в отсортированном списке.
Как pg_trgm решает эту проблему?
1.  Создание триграммного индекса: Когда ты создаешь GIN-индекс (General Inverted Index) с использованием pg_trgm, PostgreSQL для каждой строки в индексируемой колонке (например, name) разбивает её на все возможные триграммы.
2.  Хранение триграмм: В индексе хранится список всех уникальных триграмм, а для каждой триграммы — список ссылок на те строки таблицы, где эта триграмма встречается.
3.  Поиск: Когда ты делаешь запрос WHERE name LIKE '%лыжи%':
    *   PostgreSQL тоже разбивает шаблон %лыжи% на триграммы (_лы, лыж, ыжи, жи_).
    *   Затем он ищет эти триграммы в триграммном индексе.
    *   Индекс быстро выдаёт список всех строк, где эти триграммы найдены.
    *   На последнем этапе PostgreSQL перепроверяет эти найденные строки, чтобы убедиться, что полное слово "лыжи" действительно присутствует, а не просто набор триграмм.
Это значительно быстрее, чем сканировать всю таблицу, так как индекс позволяет сразу отсеять подавляющее большинство нерелевантных строк.
---
Команда CREATE EXTENSION IF NOT EXISTS pg_trgm;
Эта команда делает следующее:
*   CREATE EXTENSION: Это стандартный способ добавления дополнительной функциональности в PostgreSQL. Расширения — это заранее написанные модули, которые расширяют возможности базы данных.
*   pg_trgm: Указывает, какое именно расширение мы хотим установить.
*   IF NOT EXISTS: Это защитная часть команды. Она гарантирует, что если расширение pg_trgm уже установлено в текущей базе данных, команда не вызовет ошибку, а просто проигнорируется. Если расширения нет, оно будет установлено.
Важно: Расширения устанавливаются для конкретной базы данных. Если у тебя несколько баз данных в PostgreSQL, и ты хочешь использовать pg_trgm в каждой из них, тебе нужно будет выполнить эту команду в каждой базе данных.
---
Плюсы и минусы pg_trgm:
Плюсы:
*   Ускоряет поиск подстрок: Значительно повышает производительность запросов с LIKE '%xxx%', ILIKE '%xxx%', а также другими операторами подобия.
*   Гибкость: Позволяет находить не только точные вхождения, но и похожие слова (что полезно для автодополнения, исправления опечаток).
*   Простота использования: Легко устанавливается и интегрируется с SQL-запросами.
Минусы:
*   Размер индекса: Триграммные индексы (особенно GIN-индексы) могут быть довольно большими и занимать много места на диске, потому что для каждой строки генерируется много триграмм.
*   Время создания индекса: Создание такого индекса может занять много времени, особенно на больших таблицах.
*   Накладные расходы на запись: Любые операции INSERT/UPDATE/DELETE в индексируемой колонке будут требовать обновления триграммного индекса, что может немного замедлить эти операции.
*   Не панацея: Для очень сложных полнотекстовых задач (с учетом морфологии, синонимов, ранжирования по релевантности) всё ещё лучше использовать полноценные системы полнотекстового поиска (например, встроенный Full-Text Search в PostgreSQL на основе tsvector/tsquery или внешние системы типа Elasticsearch).
В твоем случае, pg_trgm был идеальным решением для ускорения LIKE '%лыжи%', так как B-tree был бесполезен, а pg_trgm позволил базе данных использовать индексный поиск вместо полного сканирования таблицы.








"Sargable" (сокращение от Search Argumentable) — это когда условие в твоём запросе (часть WHERE) может использовать индекс для быстрого поиска данных.
Проще говоря, если условие "sargable", то база данных может эффективно использовать индекс, чтобы найти нужные строки, не просматривая всю таблицу. Если условие "не sargable", то база данных вынуждена проверять КАЖДУЮ строку в таблице, даже если есть индекс.


Почему upper(city) не Sargable?
Представь, у тебя есть индекс на колонке city. В этом индексе города хранятся так: "Москва", "Санкт-Петербург", "новосибирск".
Индекс отсортирован по оригинальному значению.
Твой запрос: WHERE upper(city) LIKE '%СК%'
Что происходит:
1.  PostgreSQL смотрит на индекс city: Он отсортирован. Но условие upper(city) означает, что каждое значение city сначала нужно преобразовать к верхнему регистру, и только потом сравнивать.
2.  Проблема: Индекс не содержит значения upper(city). Он содержит только city. PostgreSQL не может использовать отсортированный индекс для поиска upper('Москва'), потому что upper() меняет значение.
3.  Решение PostgreSQL: Поскольку индекс бесполезен, база данных решает: "Окей, я не могу использовать индекс. Мне придётся прочитать КАЖДУЮ строку в таблице users (Seq Scan), для каждой строки вызвать функцию upper(city), и только потом сравнить результат с '%СК%'."







Bitmap Index Scan — это когда база сначала отмечает «биты» (адреса) нужных строк в индексе, а потом одним махом идет в таблицу за данными.

Запрос 2: ORDER BY + LIMIT (Золотой стандарт)
Результат: Ускорение в 640 раз (с 118 мс до 0.1 мс).
Почему это магия: Это самый важный момент. Без индекса базе нужно: 1. Прочитать всё. 2. Отсортировать миллион строк в памяти. 3. Выкинуть всё, кроме первых 10. С индексом база знает: индекс уже отсортирован. Она просто берет первые 10 записей из «картотеки» и всё. Ей не нужно смотреть остальные 999 990 строк.

Запрос 3: B-tree vs Hash
B-tree (Дерево): Самый частый индекс. Он как дерево: «Больше этого? Иди направо. Меньше? Налево». Универсален.
Hash: Работает как словарь. «Где ключ 'Завершен'? Вот по этому адресу».
Результат: Hash оказался быстрее на 30%.
Вывод: Для поиска по точному значению (=) Hash лучше, но он бесполезен, если ты захочешь найти цены BETWEEN (диапазон).

Запрос 4: LIKE '%лыжи%' (Текстовый поиск)
Ловушка: Обычный индекс (B-tree) может найти слова на букву «Л» (LIKE 'лыжи%'), но он беспомощен, если % стоит в начале. Почему? Потому что алфавитный указатель не поможет найти слова, заканчивающиеся на «...лыжи».
Решение: Индекс trigram (GIN). Он разрезает слова на кусочки по 3 буквы (триграммы).
Результат: Индекс заработал там, где обычный не мог.
Запрос 5: Соединение таблиц (JOIN)
Шок-результат: Стало медленнее на 65%.
Объяснение для препода (самое важное!): «Индексы — это не всегда хорошо». У нас уже были первичные ключи (Primary Keys). По ним PostgreSQL делал Merge Join (очень быстрый способ склейки двух отсортированных списков). Когда мы добавили лишний индекс, планировщик «запутался» и выбрал Nested Loop (вложенный цикл), который на таком объеме данных оказался хуже. Вывод: не лепи индексы куда попало.
Запрос 6: Негативный сценарий (Функции убивают индексы)
Ситуация: Мы ищем UPPER(city).
Результат: Индекс на поле city проигнорирован.
Почему: Это называется SARGABLE (Search ARGument ABLE). Индекс хранит слова «Москва», «Сочи». А ты просишь найти «МОСКВА». База говорит: «У меня в указателе нет слов капсом, я не могу их сравнить напрямую. Придется читать всё и для каждой строки вычислять функцию UPPER».
Вывод: Чтобы это работало, нужен «индекс по выражению».




Препод спросит: «А почему бы нам не проиндексировать вообще все колонки?»
Твой ответ:
«За каждый индекс мы платим скоростью записи. В моем тесте INSERT замедлился на 11%. Почему? Потому что когда мы вставляем строку, базе нужно не только записать её в таблицу, но и обновить все 5-10 индексов, которые мы создали. Каждый индекс — это отдельный файл/структура, которую нужно перестраивать при каждом изменении данных».




Если тебя спросят: «Какой самый главный вывод из твоей работы?»
Отвечай: «Самый быстрый индекс тот, который позволяет избежать сортировки (ORDER BY) и полного сканирования таблицы (Seq Scan). Но важно помнить, что индексы — это компромисс между скоростью чтения и скоростью записи, и их нужно подбирать под конкретные запросы, а не создавать вслепую».








Вот четкое техническое определение, которое примет любой преподаватель:
Индекс — это вспомогательная структура данных, связанная с таблицей, которая предназначена для ускорения поиска, фильтрации и сортировки записей.
Техническая суть (как это работает внутри):
Хранение: Индекс хранит значения одного или нескольких столбцов таблицы в упорядоченном виде.
Связь: Каждому значению в индексе соответствует указатель (физический адрес или идентификатор) на конкретную строку в основной таблице.
Алгоритм: Вместо полного последовательного сканирования таблицы (Sequential Scan), СУБД использует эффективные алгоритмы поиска по индексу (например, бинарный поиск в B-дереве), чтобы мгновенно получить адреса нужных строк и обратиться к ним напрямую.
Цена использования:
Дисковое пространство: Индексы — это отдельные файлы или структуры, требующие дополнительного места.
Производительность записи: Любая операция изменения данных (INSERT, UPDATE, DELETE) замедляется, так как СУБД обязана одновременно обновлять и саму таблицу, и все связанные с ней индексы.
Если препод спросит кратко:
«Это отсортированный набор ссылок на строки таблицы, который позволяет СУБД находить данные по логарифмическому времени вместо линейного».








«правило левого префикса»
В твоем случае (price_per_day, category_id, total_quantity) данные в индексе выстраиваются в лексикографическом порядке:
Сначала всё сортируется по цене (price_per_day).
Если цены одинаковые, эти записи сортируются внутри «ценовой группы» по категории (category_id).
Если и цены, и категории одинаковые, тогда они сортируются по количеству (total_quantity).


В узлах дерева теперь хранятся не одиночные числа, а кортежи (наборы значений).
При поиске база сравнивает твой запрос с этими наборами.
Если ты ищешь price = 1000, база быстро находит ветку, где начинаются тысячи.
Если ты ищешь price = 1000 AND category = 3, она идет в ветку тысячи и внутри неё ищет тройку.

Поскольку данные отсортированы «вложенно», композитный индекс не всегда эффективен.
Запрос по price: Индекс работает идеально (это первый уровень сортировки).
Запрос по price и category: Работает идеально (первый и второй уровни).
Запрос ТОЛЬКО по category: Индекс не сработает (или будет крайне неэффективен).
Почему? Потому что категории разбросаны внутри разных цен. Это всё равно что искать в телефонном справочнике всех людей по имени «Иван», не зная фамилий. Тебе всё равно придется пролистать весь справочник от начала до конца.



Препод спросит: «Почему база перестала параллелить запрос после создания индекса?»
Твой ответ:
«Параллельное сканирование имеет накладные расходы (overhead) на запуск процессов и сборку данных. Когда появился индекс, объем работы сократился настолько (мы теперь читаем не миллион строк, а всего несколько тысяч), что базе стало выгоднее и быстрее выполнить запрос в одном потоке, чем тратить время на координацию нескольких воркеров».


Да! В PostgreSQL существуют такие операции, как:
Parallel Index Scan (Параллельное сканирование индекса).
Parallel Index Only Scan (Параллельное сканирование только индекса).
Если таблица будет не 1 миллион, а, допустим, 100 миллионов строк, и твой фильтр выберет не 60 тысяч записей, а 10 миллионов, то планировщик вполне может решить: «Окей, индекс — это круто, но 10 миллионов строк даже по индексу читать долго. Запущу-ка я несколько воркеров, чтобы они параллельно читали разные части этого индекса».






Давай разберем Bitmap Heap Scan. Это один из самых умных алгоритмов в PostgreSQL. Он включается тогда, когда обычный Index Scan уже невыгоден, а Sequential Scan всё еще слишком дорог.
Это «золотая середина» для случаев, когда нужно достать из таблицы не 1-10 строк, а, например, как в твоем случае, 60 000 из миллиона (примерно 5-10% данных).


Когда ты ищешь по обычному индексу, база делает так:
Находит в индексе адрес первой строки -> Прыгает на диск в таблицу -> Читает строку.
Находит адрес второй строки -> Прыгает на диск -> Читает строку.
Если данные в таблице на диске лежат хаотично (не в том порядке, что в индексе), головка жесткого диска будет метаться туда-сюда (это называется Random I/O). Это очень медленно. Более того, база может прочитать одну и ту же страницу диска 10 раз, потому что на ней лежат 10 нужных строк, найденных в разное время.



Чтобы не прыгать по диску как сумасшедший, PostgreSQL разбивает работу на два шага:
Шаг 1: Bitmap Index Scan (Создание карты)
База идет в индекс и выписывает адреса всех подходящих строк. Но она не бежит сразу в таблицу! Она создает в памяти битовую карту (Bitmap).
Представь таблицу как набор блоков (страниц) на диске.
Битовая карта — это массив нулей и единиц.
База ставит «1» напротив каждой страницы диска, где индекс нашел хотя бы одну нужную строку.
Результат первого шага: У нас есть «карта сокровищ», где отмечены все блоки таблицы, которые нам интересны.
Шаг 2: Bitmap Heap Scan (Посещение таблицы)
Теперь база берет эту карту и начинает читать таблицу (Heap).
Главная фишка: Она читает блоки строго по порядку, как они лежат на диске.
Она заходит в блок, забирает оттуда все нужные строки и никогда не возвращается в него второй раз.



Bitmap Index Scan: Показывает, сколько времени ушло на поиск в индексе и построение карты.
Bitmap Heap Scan: Показывает время, потраченное на чтение самих данных из таблицы по карте.
Recheck Cond: Поскольку в битовой карте «1» может ставиться на целую страницу (где лежит много строк), базе нужно еще раз проверить условие фильтра для каждой строки внутри этого блока, чтобы не отдать лишнего.
Heap Blocks: exact=3708: Это значит, что база посетила 3708 физических блоков на диске.


«Bitmap Heap Scan — это двухэтапный алгоритм доступа. Сначала через Bitmap Index Scan строится битовая карта страниц данных, на которых индекс нашел совпадения. Затем на этапе Bitmap Heap Scan база читает эти страницы в порядке их физического расположения на диске.
База выбрала этот метод вместо обычного Index Scan, потому что количество выбираемых строк (около 65 000) достаточно велико. Обычный Index Scan привел бы к огромному количеству хаотичных прыжков по диску (Random I/O) и многократному чтению одних и тех же страниц. Bitmap Scan позволил упорядочить чтение и посетить каждую нужную страницу таблицы ровно один раз».


ожет ли быть и Bitmap, и параллелизм одновременно?
Да! Начиная с PostgreSQL 10, существует алгоритм Parallel Bitmap Heap Scan.
Это выглядит как настоящий «спецназ» в твоем плане выполнения. Давай разберем, как они делят работу в этом случае:
Этап подготовки (Leader): Основной процесс (лидер) идет в индекс и строит ту самую битовую карту (Bitmap Index Scan). Он один отмечает: «Так, нам нужны блоки №5, №10, №100... №100000».
Этап выполнения (Workers): Лидер отдает эту карту воркерам. И вот тут начинается магия параллелизма:
Воркер 1 берет первую часть карты и читает блоки №5, №10...
Воркер 2 берет вторую часть карты и одновременно читает блоки №500, №510...
Они читают таблицу параллельно, но каждый из них внутри своего куска движется упорядоченно по карте, не прыгая по диску.







Если ты ищешь по префиксу (когда % только в конце, например 'лыжи%'), B-tree — это абсолютный чемпион. На защите это отличный вопрос, чтобы показать, что ты понимаешь внутреннее устройство индексов.
Давай разберем, почему так происходит:
1. Как работает B-tree в этом случае (Аналогия со словарем)
B-tree хранит данные в отсортированном виде.
Когда ты пишешь LIKE 'лыжи%', для базы это превращается в поиск по диапазону: «Дай мне всё, что больше или равно 'лыжи' и меньше, чем следующая буква по алфавиту».
Как это делает B-tree: Он открывает «словарь» на странице, где начинаются слова на «Лы...», и просто читает строки одну за другой, пока слова не закончатся. Ему нужно сделать всего несколько прыжков по дереву. Это максимально эффективно.
2. Как работает Trigram (GIN)
Тримграмный индекс (GIN) — это инвертированный индекс. Он хранит не сами слова, а список соответствий: «Триграмма 'лыж' встречается в строках №5, №10, №100».
Как это делает Trigram: Когда ты ищешь 'лыжи%', он разрезает твой запрос на триграммы (лыж, ыжи). Затем он лезет в индекс, достает списки ID для каждой триграммы и начинает их пересекать (искать ID, которые есть в обоих списках). После этого он все равно идет в таблицу перепроверять результат.
Итог: Это намного больше вычислительной работы, чем просто прочитать кусок отсортированного списка в B-tree.



1. Как он «нарезает» данные (Триграммы)
Когда ты вставляешь слово в таблицу, расширение pg_trgm разрезает его на кусочки по 3 символа (триграммы).
Возьмем слово «Лыжи»:
База добавляет пробелы в начале и конце: лыжи
Режет на куски: л, л ы, лыж, ыжи, жи... (каждый кусок — это 3 символа).
2. Как устроен индекс (Inverted Index)
Внутри индекса хранится огромный список всех уникальных триграмм, которые когда-либо встречались в твоей таблице. И для каждой триграммы записан список адресов строк, где она есть.
Это выглядит так:
Триграмма лыж -> встречается в строках [№1, №5, №10, №125...]
Триграмма ыжи -> встречается в строках [№1, №8, №10, №300...]
Триграмма сол -> встречается в строках [№2, №15...]
3. Как происходит поиск (например, LIKE '%лыжи%')
Когда ты запускаешь запрос, база делает следующее:
Берет твое слово «лыжи» и тоже режет его на триграммы: лыж и ыжи.
Идет в индекс и достает два списка:
Список для «лыж»: [1, 5, 10, 125]
Список для «ыжи»: [1, 8, 10, 300]
Пересекает их (выполняет операцию логического И). Она ищет только те ID, которые есть в обоих списках.
Получает результат: [1, 10].
4. Зачем нужен Recheck (проверка)?
Триграммы — это «шумный» способ поиска. Например, слово «Лыжня» тоже содержит триграммы лыж и ыжи.
Поэтому в плане выполнения (EXPLAIN ANALYZE) ты всегда видишь Rows Removed by Index Recheck. База нашла «кандидатов» по индексу, сходила в таблицу, прочитала их и проверила: «Там реально написано 'лыжи' или что-то другое?».







1. Почему в твоем случае стало медленнее?
В твоем запросе №5 база склеивала 4 таблицы. До того, как ты создал свой индекс, у базы уже были Первичные ключи (Primary Keys).
Как база работала ДО (Merge Join):
Первичные ключи (id) — это уже отсортированные индексы. Когда ты делаешь JOIN users u ON r.renter_id = u.id, PostgreSQL видит: «Ага, таблица users идеально отсортирована по id. Я просто возьму список из rentals и очень быстро сопоставлю его с users как две половинки молнии на куртке». Это называется Merge Join. Он работает невероятно быстро на больших объемах.
Что произошло ПОСЛЕ (Nested Loop):
Ты создал новый индекс idx_rentals_renter_status. Планировщик PostgreSQL — это программа, которая пытается угадать самый быстрый путь. Он увидел твой новый индекс и «подумал»:
«О, тут есть индекс по статусу и арендатору! Дай-ка я попробую использовать его».
Он переключил стратегию на Nested Loop (вложенный цикл).
Nested Loop работает так: он берет одну строку из индекса, прыгает в таблицу rentals, забирает данные, потом прыгает в таблицу users, и так 1000 раз (из-за твоего LIMIT 1000).
Итог: Вместо того чтобы быстро «проскользнуть» по уже готовым первичным ключам (последовательное чтение), база начала совершать тысячи мелких прыжков по диску (Random I/O) туда-сюда между новым индексом и таблицами. Эти «прыжки» и сожрали лишнее время.
2. Всегда ли так получается? (Нет!)
Индексы для JOIN — это база производительности. Если бы у тебя не было первичных ключей или внешних ключей (FK), твой запрос без индексов выполнялся бы не 27 мс, а несколько минут, потому что базе пришлось бы делать полный скан (Seq Scan) всех четырех таблиц для каждой строки.
Когда индекс для JOIN ОБЯЗАТЕЛЕН:
Если ты соединяешь таблицы по полю, которое не является первичным ключом и по нему нет индекса.
Например: Ты соединяешь users и rentals не по id, а по какому-нибудь email или phone, который ты забыл проиндексировать. Без индекса это будет катастрофа.
3. Золотые правила: как использовать индексы с JOIN
Чтобы не наступить на те же грабли, запомни три правила для защиты:
Индексируй Внешние ключи (Foreign Keys): Почти всегда стоит иметь индекс на полях, по которым идет соединение (renter_id, equipment_id). В твоем случае база использовала системные индексы первичных ключей, поэтому твой ручной индекс оказался лишним.
Доверяй, но проверяй (EXPLAIN): Как ты увидел в лабе, база не всегда выбирает лучший путь. Иногда создание индекса «сбивает прицел» планировщику, и он выбирает плохую стратегию (как Nested Loop вместо Merge Join).
Индекс должен сужать поиск: Индекс хорош, когда он помогает из миллиона строк сразу выкинуть 999 тысяч. Если твой фильтр (status = 'Завершен') всё равно оставляет 200 тысяч строк, индекс по этому полю может быть менее эффективен, чем простое чтение таблицы подряд.






Почему такая гигантская разница?
Любой UPDATE состоит из двух этапов:
Поиск (Find): База должна найти строку, которую нужно изменить.
Изменение (Modify): База записывает новые данные на диск.
Сценарий 1: По индексированному полю (id)
База использует Index Scan. Она не читает миллион строк. Она делает 3–5 прыжков по B-дереву (как по оглавлению книги), находит физический адрес строки на диске и мгновенно её меняет.
Время поиска: почти 0.
Итог: 0.247 мс.
Сценарий 2: По неиндексированному полю (price_per_day)
База вынуждена делать Seq Scan. У неё нет «оглавления» для цен. Она обязана прочитать все 1 000 000 строк с диска в оперативную память, проверить каждую: «Твоя цена равна X?», и только если совпало — обновить.
Время поиска: чтение всего миллиона строк с диска. Это и занимает те самые 200 мс.
Итог: 202.271 мс.



Тонкий момент для защиты (Важно!)
Препод может спросить: «А если я обновляю само поле, по которому есть индекс, это ускорит или замедлит UPDATE?»
Правильный ответ:
Если индекс помогает найти строку (в блоке WHERE), он дает колоссальное ускорение (как в твоем примере).
Но если ты меняешь значение в поле, на котором стоит индекс (например, SET price_per_day = 100 и у тебя есть индекс на цену), то сам процесс записи станет чуть-чуть медленнее.
Почему? Потому что базе нужно не только изменить строку в таблице, но и найти старое значение в индексе, удалить его и вставить новое в правильное место B-дерева.
